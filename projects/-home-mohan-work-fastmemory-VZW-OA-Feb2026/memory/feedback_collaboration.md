User wants structured, checkpoint-based collaboration: decompose first, then tradeoff analysis, then recommendation. Don't jump to code.

**Why:** User is an experienced architect who wants to think through design decisions collaboratively before implementation. They ask probing questions to validate architecture (e.g., "how exactly does RAG fit?", "how do you handle cross-doc?") and expect precise, honest answers.

**How to apply:**
- Always align on architecture before writing code
- When user asks "how does X work in your proposal?" — give a concrete, traced example, not a hand-wavy explanation
- Surface alternatives and tradeoffs proactively
- Flag when I'm uncertain rather than presenting false confidence
- When user says "let's align first" — stop and discuss, don't start building
