## Why 8?

In x86-64, a `long` is **64 bits = 8 bytes**. Memory is byte-addressed, so to get from element `i` to element `i+1`, you have to jump 8 bytes forward.

```

[0x00] [0x08] [0x10] [0x18]   ← addresses (hex)
```

So `arr[i]` in C silently compiles to `base_address + i * sizeof(long)` which is `base + i * 8`.

## The Assembly Has No Type System

When the CPU executes:
```asm
imulq  $8, %rcx
```
It has **zero idea** you're working with longs. It just sees:
- A register containing the number `i`
- An instruction to multiply by 8
- A memory dereference at the result

The CPU only deals in raw bytes and addresses. **The `* 8` is the programmer's responsibility**, not the CPU's. If this were an array of `int` (4 bytes), you'd multiply by 4. `char` array? Multiply by 1 (i.e., no multiply needed).

## How a Compiler Handles This

When you write C:
```c

```
The compiler sees the type `long*`, knows `sizeof(long) == 8`, and automatically emits the `* 8`. You never write it yourself in C. But dropping down to assembly means **you are the compiler** — you have to encode that type knowledge manually.

When the CPU decides it needs something from RAM, it doesn't just fetch that one value. Next time you access anything in that same 64-byte block, it comes from cache (~0.3ns) instead of RAM (~100ns).

The cache line size is the same across all levels, but the **total size** and **speed** differs:

  L1 Cache  →  32-64 KB,   ~4 cycles    (per core, fastest)

4 cycles    12 cycles   40 cycles   200 cycles

## How They Work Together

When the CPU loads a cache line it goes into L1. L2 and L3 act as a **spillover hierarchy**:

```
L1 fills up → oldest lines get demoted to L2
L2 fills up → oldest lines get demoted to L3
L3 fills up → oldest lines get evicted to RAM entirely
```

And on a cache miss the reverse happens — the line gets promoted up from wherever it currently lives:

```
L1 miss → found in L2 → copy up to L1, takes ~12 cycles
L1 miss → L2 miss → found in L3 → copy up to L1/L2, takes ~40 cycles
L1 miss → L2 miss → L3 miss → go to RAM, takes ~200 cycles
```

Assistant:  ## The Problem: Registers Get Clobbered

When you call any function, that function will use the argument registers (`%rdi`, `%rsi`, `%rdx`, etc.) for its own purposes. So if you stored something important in `%rsi` before the call, it's gone after:

```asm
movq  $42, %rsi      # store important value in rsi
call  printf         # printf uses rsi internally
                     # rsi now has garbage, your 42 is gone
```

---

## Two Solutions

### Solution 1: Push/Pop Around the Call
Save the registers onto the stack before the call, restore them after:

```asm
movq  $42, %rsi
pushq %rsi           # save rsi to stack
call  printf
popq  %rsi           # restore rsi from stack
                     # rsi = 42 again
```

The catch — you must push in **pairs** to maintain 16-byte stack alignment:

```asm
pushq %rsi           # 8 bytes
pushq %rdi           # 8 bytes  → total 16, aligned
call  printf
popq  %rdi
popq  %rsi
```

If you only push one register (8 bytes), the stack is misaligned and you'll crash.

---

### Solution 2: Use Callee-Saved Registers

This is the cleaner solution shown in the code. Instead of using argument registers (`%rdi`, `%rsi` etc.) to store your values, use **callee-saved registers** like `%r13`, `%r14`, `%r12`, `%rbx`.

These registers are special because **by convention, any function that uses them must restore them before returning**. So if printf internally uses `%r13`, it will push and pop it itself before returning — your value is safe.

```asm

    pushq %rbp
    movq  %rsp, %rbp
    pushq %r13           # save old r13 value (we're about to use it)
    pushq %r14           # save old r14 value — pushed in pairs for 16 byte alignment

    # now freely use %r13 and %r14 to store your values
    # even if you call printf/scanf inside, they won't touch these
    # or if they do, they'll restore them before returning

    popq  %r14           # restore in REVERSE order
    popq  %r13
    leave
    ret
```

```asm

                     # r14 still has i, untouched
```

User: explain
Assistant:  ## Static vs Shared Libraries

This slide introduces two types of libraries and how the linker handles them differently.

---

## Static Libraries (.a files)

The linker takes the library code and **physically copies it into your executable**:

```
main.o + libmath.a  →  executable (contains everything, self contained)
```

Your executable is bigger but **completely self contained**. It can run on any machine with no dependencies. This is why some programs "just work" when you copy them to another machine.

---

## Shared Libraries (.so files)

The linker does NOT copy the code in. It just **verifies the symbols exist** and leaves a reference:

```
main.o + libc.so  →  executable (just has "I need printf from libc.so")
```

The actual library code stays separate. At **runtime** when you actually run the program, the OS loads the shared library into memory and connects everything up.

```
runtime:
executable  +  libc.so loaded by OS  →  program runs
```

---

## Why The Distinction Matters

| | Static (.a) | Shared (.so) |
|---|---|---|
| Library copied into executable | Yes | No |
| Executable size | Bigger | Smaller |
| Dependencies at runtime | None | Needs .so file present |
| Memory usage | Each program has own copy | All programs share one copy in memory |
| Example | your own utility library | libc, libpthread |

---

```

```

```

This is the linker telling you — I searched every `.o` file and every library you gave me, and I cannot find this symbol anywhere. It's one of the most common errors in C development and it always means either you forgot to link a library or you have a typo in the function name.

The linker is essentially doing three things:

User: explain the memory usage tradeoff between shared and static in the table, and also, does the user have to specify if the library they want to create is to be shared or static?
Assistant:  ## Memory Usage Tradeoff

**Static — every program gets its own copy:**

```
Program A running:    [A's code + printf copy 1 + malloc copy 1]
Program B running:    [B's code + printf copy 2 + malloc copy 2]
Program C running:    [C's code + printf copy 3 + malloc copy 3]
```

If 100 programs are running that all use printf, you have **100 copies of printf in RAM simultaneously**. Complete waste.

**Shared — everyone points to the same copy:**

```
Program A running:    [A's code] ──┐
Program B running:    [B's code] ──┤──→ [one copy of libc.so in RAM]
Program C running:    [C's code] ──┘
```

The OS loads libc.so into memory **once**, and every program that needs it just points to that same copy. 100 programs running = still just 1 copy of printf in RAM.

This is literally why libc is a shared library — virtually every C program on your system uses it, so sharing one copy saves enormous amounts of RAM.

Yes, you explicitly tell the compiler/linker which you want.

**Creating a static library:**
```bash

```

**Creating a shared library:**
```bash
gcc -c -fPIC mylib.c -o mylib.o    # compile with Position Independent Code flag

```

The `-fPIC` flag is important for shared libraries — it tells the compiler to generate code that can be loaded at **any memory address**, since the OS decides where to load the shared library at runtime and it might be different every time.

**Linking against them:**
```bash
gcc main.c -L. -lmylib          # linker decides based on what it finds
gcc main.c -L. -static -lmylib  # force static linking
```

User: how much overhead does shared libraries use, and why are windows shared libraries .dll
Assistant:  ## Shared Library Overhead

There are two costs:

**1. Load time overhead** — when your program starts, the OS has to find and load the `.so` files into memory and resolve all the symbol addresses:

```
program starts
    → OS finds libc.so
    → loads it into memory (if not already loaded)
    → patches all the call addresses
    → program actually starts running
```

This adds maybe **a few milliseconds** to startup time. Negligible for most programs but noticeable for tiny command line tools that run thousands of times.

**2. Runtime call overhead (PLT/GOT)** — every call to a shared library function goes through an extra layer of indirection called the **Procedure Linkage Table**:

```
your code calls printf
    → jumps to PLT entry        # extra indirection
    → PLT looks up real address in GOT (Global Offset Table)
    → jumps to actual printf
```

Static call:
```
your code calls printf → jumps directly to printf
```

The PLT/GOT indirection costs roughly **1-3 nanoseconds extra per call**. Essentially nothing in practice.

---

## In Reality

The overhead is so small that the benefits almost always outweigh it:

| Cost | Reality |
|---|---|
| Load time | Few milliseconds at startup, one time cost |
| Per call overhead | ~1-3 nanoseconds, basically nothing |
| Memory savings | Enormous — one copy shared across all programs |

## Why Windows Uses .dll

`.dll` stands for **Dynamic Link Library** — it's Microsoft's equivalent of `.so`, just a different name for the same concept.

The reason they're different is purely historical:

**Unix/Linux** was designed in the early 1970s at Bell Labs with a philosophy of open standards and portability. They called them shared objects (`.so`) reflecting what they technically are — shared object files.

**Windows** was designed independently by Microsoft in the 1980s-90s with no connection to Unix. Microsoft called them Dynamic Link Libraries (`.dll`) reflecting how they're used — dynamically linked at runtime.

## And the Sharing Actually Happens Here

If program A and program B both need libc.so:

```
Program A memory:   [A's code] [libc.so mapped here]
                                      ↑
                               same physical RAM
                                      ↓
Program B memory:   [B's code] [libc.so mapped here]
```

The OS maps the **same physical memory pages** of libc.so into both processes' virtual address spaces. Both programs see libc at potentially different virtual addresses, but it's the same actual RAM underneath. This is the shared library memory saving in action — one physical copy, used by everyone.

VIRTUAL ADDRESS SPACE          PHYSICAL RAM

Program B:                                  SAME PHYSICAL RAM
[0x400000 - B's code    ]                       ↑
[0x8f000 - libc.so      ]─────────────→ [physical page 0x1000 - libc code]
```

Each process **thinks** it has its own private copy of libc at its own address. But the OS is secretly pointing both of them at the same physical RAM page underneath.

---

## Why Virtual Address Spaces Exist

This is the whole point of virtual memory — every process gets the illusion of its own private address space:

```
Program A thinks it owns all of memory from 0x0 to 0xffffffff
Program B thinks it owns all of memory from 0x0 to 0xffffffff
```

They don't actually — the OS manages a mapping table (page table) that translates each process's virtual addresses to actual physical RAM locations. Two different virtual addresses in two different processes can point to the same physical RAM.

---

## The One Exception — Writable Data

The sharing only works for **read only code**. If a shared library has writable data (global variables inside the library), the OS uses a technique called **copy on write**:

```
initially:     A and B both point to same physical page
program A writes to library's global variable
OS intercepts: makes a private copy of that page just for A
now:           A has its own modified copy, B still has original
```

So code pages are truly shared, data pages get privately copied the moment anyone writes to them.

## Every Single Memory Access Goes Through This

It's not just shared libraries — absolutely everything gets translated:

```
your code accesses any address
    │
    ▼
CPU's Memory Management Unit (MMU)
    │
    │  looks up page table
    ▼
physical RAM address
    │
    ▼
actual data
```

The MMU is dedicated hardware inside the CPU that does this translation on every single memory access, so fast it's essentially invisible.

## The Page Table is What the OS Controls

This is how the OS enforces memory isolation between processes:

```
Program A tries to access Program B's memory
    │
    ▼
MMU looks up A's page table
    │
    ▼
that address isn't mapped in A's page table
    │
    ▼
segfault — access denied
```

Program A literally cannot see Program B's private memory because the OS never put those mappings in A's page table. The hardware enforces it automatically on every access.

## The Problem — Indexing Requires Multiplication

As we discussed earlier, every array access requires computing:

```c
arr[i]  →  base_address + i * sizeof(element)
```

For an `int` array that's `i * 4`. Multiplication is one of the more expensive CPU operations compared to addition or bit shifting.

---

## The Optimization — Bit Shifting

Multiplying by a power of 2 is identical to a left bit shift:

```
i * 1  ==  i << 0
i * 2  ==  i << 1
i * 4  ==  i << 2    ← int array
i * 8  ==  i << 3    ← long array
i * 16 ==  i << 4
```

Bit shifting is **much faster** than multiplication — it's a single cycle operation vs multiple cycles for `imulq`. So the compiler will automatically replace:

```asm
imulq  $4, %rcx     # slow — actual multiplication
```
with:
```asm
salq   $2, %rcx     # fast — shift left by 2 (same result, cheaper)
```

---

## The Catch — Only Works for Powers of 2

This optimization only applies when element size is a power of 2. Standard types all happen to be powers of 2:

```
char   = 1 byte  = 2^0  ✓
short  = 2 bytes = 2^1  ✓
int    = 4 bytes = 2^2  ✓
long   = 8 bytes = 2^3  ✓
```

But if you have a struct:

```c
struct Thing {
    int a;      // 4 bytes
    int b;      // 4 bytes
    char c;     // 1 byte
};              // 9 bytes total — NOT a power of 2
```

```c
arr[i]  →  base + i * 9
```

You can't bit shift by 9. The compiler is forced to use actual `imulq` multiplication, which is why the slide says **integer multiplication may be needed**.

A "string array" in C is actually an **array of pointers**, each pointing to a string somewhere in memory:

## What an OS Actually Needs

The OS kernel itself is written in C but uses almost **none of the standard C library**. The kernel can't use printf, malloc, or any libc functions because:

```
libc depends on the OS to function
OS kernel is what libc calls INTO
```

It's circular — libc calls the OS for memory, I/O, etc. The kernel IS the bottom of the stack, there's nothing below it to call.

So the kernel implements its own versions of everything it needs:

```c
// linux kernel has its own:
printk()       // instead of printf
kmalloc()      // instead of malloc
memcpy()       // reimplemented from scratch
strlen()       // reimplemented from scratch
```

---

## The Layers

```
user programs          → use libc (printf, malloc, etc.)
        ↓ calls
libc                   → translates to system calls
        ↓ calls
OS kernel              → handles actual hardware
        ↓ talks to
hardware
```

---

## Who Provides libc Then?

When building a complete OS, someone has to provide libc for user programs. The options are:

**Port an existing libc** — most OS developers just port an existing implementation:
- **glibc** — GNU's libc, used on Linux
- **musl** — lightweight alternative libc
- **newlib** — common for embedded/hobby OSes
- **bionic** — Android's custom libc

**Write your own** — some do write their own minimal libc, especially for embedded or hobby OSes where they want full control or minimal size.

**Use a language runtime** — some modern OSes (like some research OSes) skip C entirely and use Rust or other languages with their own standard libraries.
